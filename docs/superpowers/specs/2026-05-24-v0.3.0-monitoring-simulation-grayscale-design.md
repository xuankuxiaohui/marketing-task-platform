## 1. Event Tracking（埋点事件）

### 1.1 event_log 表 (Flyway V10)

```sql
CREATE TABLE event_log (
    id            BIGINT       AUTO_INCREMENT PRIMARY KEY,
    event_type    VARCHAR(32)  NOT NULL,
    task_id       BIGINT       DEFAULT NULL,
    instance_id   BIGINT       DEFAULT NULL,
    step_id       BIGINT       DEFAULT NULL,
    user_id       VARCHAR(64)  DEFAULT NULL,
    platform      VARCHAR(16)  DEFAULT NULL,
    event_data    JSON         DEFAULT NULL,
    created_at    DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_event_type  (event_type),
    INDEX idx_task_id     (task_id),
    INDEX idx_instance_id (instance_id),
    INDEX idx_created_at  (created_at)
);
```

### 1.2 事件类型

| Event Type | 触发位置 | 携带数据 |
|---|---|---|
| `TASK_VIEWED` | `ClientTaskController.list()` 每条可见任务 | taskId, userId, platform |
| `INSTANCE_CREATED` | `TaskService.getOrCreateInstance()` 新建时 | taskId, instanceId, userId |
| `STEP_COMPLETED` | `StepAdvanceEngine.completeStep()` | taskId, instanceId, stepId, stepType |
| `REWARD_TRIGGERED` | `RewardStepHandler.onStepEnter()` | taskId, instanceId, stepId, rewardType |
| `REWARD_SUCCESS` | `RewardService.reward()` 成功返回 | taskId, instanceId, stepId, rewardType |
| `REWARD_FAILURE` | `RewardService.reward()` 异常捕获 | taskId, instanceId, stepId, error |
| `CLAIM_SUCCESS` | `ClaimService.claim()` 成功 | taskId, instanceId, prizeId, tradeNo |
| `FILTER_EVALUATED` | `FilterExpressionEngine.evaluate()` | taskId, userId, expression, result |

### 1.3 EventTrackingService

```java
@Service
public class EventTrackingService {
    private final EventLogMapper eventLogMapper;

    public void track(EventType type, Long taskId, Long instanceId,
                      Long stepId, String userId, String platform,
                      Map<String, Object> eventData);
}
```

- `track()` 内部构造 `EventLog` 实体并 `eventLogMapper.insert()`
- 调用方不感知异常（内部 try-catch，失败只 log.warn）
- 不使用异步（保持事务内一致，event_log 写入极轻量）

### 1.4 调用点注入

共 10 个调用点，每个增加 ~2 行：

| # | 位置 | 事件 |
|---|---|---|
| 1 | `ClientTaskController.list()` | TASK_VIEWED (per task in result) |
| 2 | `TaskService.getOrCreateInstance()` | INSTANCE_CREATED |
| 3 | `StepAdvanceEngine.completeStep()` | STEP_COMPLETED |
| 4 | `RewardStepHandler.onStepEnter()` | REWARD_TRIGGERED |
| 5 | `RewardService.reward()` 成功路径 | REWARD_SUCCESS |
| 6 | `RewardService.reward()` 异常路径 | REWARD_FAILURE |
| 7 | `ClaimService.claim()` 成功路径 | CLAIM_SUCCESS |
| 8 | `FilterExpressionEngine.evaluate()` | FILTER_EVALUATED |
| 9 | `InternalCallbackController.callback()` | STEP_COMPLETED (via completeStep) |
| 10 | `InternalCallbackController.progress()` | STEP_COMPLETED (via completeStep) |

```java
    // ...
}
```

Prometheus/Grafana 通过 `/actuator/prometheus` 对接（预留，当前不强制配置）。

### 2.2 task_metrics 聚合表 (Flyway V10)

用于 Admin 仪表盘日聚合查询，Micrometer 只负责实时指标：

```sql
CREATE TABLE task_metrics (
    id              BIGINT   AUTO_INCREMENT PRIMARY KEY,
    task_id         BIGINT   NOT NULL,
    metric_date     DATE     NOT NULL,
    views           BIGINT   DEFAULT 0,
    participants    BIGINT   DEFAULT 0,
    completions     BIGINT   DEFAULT 0,
    reward_success  BIGINT   DEFAULT 0,
    reward_failure  BIGINT   DEFAULT 0,
    avg_filter_ms   DOUBLE   DEFAULT 0,
    UNIQUE KEY uk_task_date (task_id, metric_date)
);
```

- `TaskMetricsScheduler`：每 5 分钟从 `event_log` 聚合写入 `task_metrics`（upsert ON DUPLICATE KEY UPDATE）
- 首次部署时回填当天数据

## 3. Admin Simulation（管理端模拟测试）

### 3.1 AdminSimulateController

```
POST /api/admin/simulate/impersonate       → 设置模拟用户上下文（返回临时 session）
POST /api/admin/simulate/callback           → 模拟 CALLBACK
POST /api/admin/simulate/progress           → 模拟 PROGRESS
POST /api/admin/simulate/full-flow/{taskId} → 一键全流程
DELETE /api/admin/simulate/impersonate      → 退出模拟
```

### 3.2 Impersonate 机制

- `POST /impersonate` body: `{ userId, province, role, tags, level, platform }`
- 写入 `SimulateContextHolder`（ThreadLocal，类似 `UserContextHolder`）
- Admin 后续调用 simulation API 时自动注入模拟上下文
- `DELETE /impersonate` 清理上下文

### 3.3 Full Flow 执行

`POST /full-flow/{taskId}` 在当前模拟用户下执行：

1. `getOrCreateInstance(task, userContext)` — 创建实例 + 级联 PASSIVE
2. 遍历步骤：CLICK → `click(instance, stepId)`
3. 遍历步骤：CALLBACK → `callback(instance, eventKey)`
4. 遍历步骤：PROGRESS → `progress(instance, stepId, targetValue)`
5. REWARD 步骤由级联自动触发
6. 返回每步执行结果列表

### 3.4 Admin 前端

任务编辑页新增「模拟测试」Tab：
- 用户身份配置面板（表单：userId, province, tags, level, platform）
- 一键全流程测试按钮
- 步骤执行进度展示（van-steps / el-steps，每步状态：pending/running/success/failed）
- Internal API 手动模拟表单（事件 key 输入 / 进度值输入）

## 4. Grayscale（任务灰度）— v0.3.1

### 4.1 task 表扩展 (Flyway V11)

```sql
ALTER TABLE task ADD COLUMN gray_type   VARCHAR(16) DEFAULT 'NONE';
ALTER TABLE task ADD COLUMN gray_config JSON DEFAULT NULL;
```

`gray_type`: `NONE` | `PERCENTAGE` | `AB` | `CROWD`
`gray_config` 示例：
```json
{"percent": 10}
{"groups": [{"name":"A","percent":50},{"name":"B","percent":50}]}
{"crowdIds": [1, 2]}
```

### 4.2 GrayService

```java
@Service
public class GrayService {
    public boolean isInGray(String userId, Long taskId, String grayType, String grayConfig);
    public String getABGroup(String userId, Long taskId, String grayConfig);
}
```

- PERCENTAGE: `Math.abs((userId + taskId).hashCode() % 100) < percent`
- AB: hash mod 100，按 groups 配置的累计百分比分配
- CROWD: 查 `list_data` 表（复用现有 ListDataService）

### 4.3 过滤函数

在 `FilterExpressionEngine.registerFunctions()` 中注册：

| 函数 | 参数 | 行为 |
|---|---|---|
| `inGrayPercent(percent)` | 10 | hash-based 百分比分流 |
| `inABGroup(groupName)` | "A" | 是否在指定 AB 组 |
| `inCrowd(crowdId)` | 1 | 是否在指定人群包 |

实现约束：
- 纯内存计算，不走 I/O（CROWD 除外，需查 DB 但允许——已有 `inAllowlist` 先例）
- hash 稳定：同一 userId+taskId 总是同一组
- `inABGroup` 需要当前 taskId 上下文，通过 ThreadLocal 传递

### 4.4 Admin 前端

任务编辑 BasicInfo 区域新增灰度配置：
- 灰度类型下拉选择器
- PERCENTAGE → 百分比滑块 (0-100)
- AB → 组名 + 百分比分配表格
- CROWD → 人群包下拉多选（list_data 中取 CROWD 类型条目）

### 4.5 灰度事件追踪

灰度决策同样通过 `EventTrackingService` 记录：
- `FILTER_EVALUATED` 事件的 `event_data` 包含 `grayType` + `grayResult`
- AB 实验时额外记录 `abGroup` 字段，用于后续 per-group 指标对比
